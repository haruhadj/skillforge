# SkillForge — Audit Report (Round 2)

*Audited: 2026-06-27 · Findings verified against the codebase before recording.*

> Follow-up to `docs/AUDIT_2026-06-26.md`. A fresh three-pronged pass (server
> security, performance/scaling, infra/deployment) surfaced a **CRITICAL regression

## CRITICAL — Fixed this round

### C1. Self-assignable `role: 'admin'` (privilege escalation) — ✅ FIXED (2026-06-27)
**File:** `firestore.rules`

**Verified:** the recursive `match /users/{uid}/{document=**}` rule granted the owner
`allow write` over **every** doc including the root profile, and `match /users/{uid}`
`allow update` had **no field guard**. Any signed-in user could run
`setDoc(doc(db,'users',myUid),{role:'admin'},{merge:true})` from the browser console and
instantly become admin — then delete any account via the (now token-gated) delete-user
route, and write `gameRegistry` / `announcements` / `config/oauthProviders` / `platform`.
This nullified round 1's fix #1, whose admin check reads the same `role` field.

**Fixed:**
- Added `isAdmin()` (reads the **caller's** own user doc, so it can't be spoofed by
  writing the target) and `noPrivilegeEscalation()` (rejects writes touching `role`).
- Replaced the broad recursive owner-write with a **subcollection-only** rule
  (`match /users/{uid}/{sub}/{document=**}`) that can never match the root doc, so
  owners keep full access to `scores` / `gameStats` (the canonical client write path)
  but cannot touch privileged root-doc fields.
- `match /users/{uid}`: `create` blocks `role: 'admin'`; `update` allows
  `(owner && roleUnchanged) || isAdmin`. All other admin-gated collections now reuse
  `isAdmin()`.
- `setUserRole()` (`app/services/adminService.ts:30`) still works because it is only
  called by an admin → passes the `isAdmin()` branch. `ensureUserProfileDocument()`
  writes no `role`, so no regression.

**Deploy:** `firebase deploy --only firestore:rules --project skillforge-7a058`.

## HIGH

### H1. Socket.IO servers: no rate limiting + unbounded room growth (DoS) — ✅ FIXED (2026-06-27)
**Files:** `server/games/{tictactoe,chess,chroma-memory}/socket-server.js`

**Verified:** `express-rate-limit` was applied only to the REST servers. The three
Socket.IO servers had no per-IP/connection throttle, and every `create_room` /
`joinRoom` did `rooms.set(...)` with no global cap and no idle TTL — a single client
could spam room creation and exhaust the container's RAM (compounded by H2).

**Fixed (all three):** added a per-IP connection throttle (`io.use`, sliding 60s
window, 30 conns/IP), a global room cap (500) + per-socket cap (5), `lastActivity`
timestamps, and a 5-min reaper `setInterval` (`.unref()`) that evicts rooms idle >30 min
(chroma also `clearRoomTimers` first) and prunes the IP table.

### H2. No container resource limits (one service can OOM the Pi) — ✅ FIXED (2026-06-27)
**File:** `docker-compose.prod.yml`

**Verified:** no `mem_limit` / `pids_limit` / `cpus` on any of the 8 services.

**Fixed:** added `mem_limit` + `pids_limit` to every service — nginx 64m, frontend 512m,
socket servers 192m each, spelling-bee/vocab 256m, hamaru 320m (in-memory JMdict).

### H3. Hamaru CORS omitted the duckdns prod origin — ✅ FIXED (2026-06-27)
**File:** `server/games/hamaru/server.js`

**Verified:** Hamaru's prod allow-list was only `https://skillforge.haruhadj.org`,
while chess/tictactoe/chroma/vocab also allow `skillforge.haruhadj.duckdns.org` →
inconsistent CORS, requests blocked if served from the duckdns origin.

**Fixed:** added `https://skillforge.haruhadj.duckdns.org` to Hamaru's prod origins.

### S2. Scores are not actually server-authoritative — OPEN (architectural)
**Files:** `firestore.rules`, `app/api/games/score/route.ts:16`, `app/services/gameDataService.ts`

`firestore.rules` lets the owner write `scores/{gameId}` + `gameStats/{gameId}` via the
client SDK, and the score route's own comment says the **canonical** in-app path *is*
that client write. So a user can `setDoc` an arbitrary `bestScore`, and the leaderboard
recompute ingests it verbatim. The S3 fix below only guards the HTTP route.
**Fix (next workstream):** migrate `PlayGameClient` to POST `/api/games/score`, then set
`scores`/`gameStats` to `allow write: if false` (Admin-SDK-only).

### P1/P2. Admin analytics/dashboard run full client-side Firestore scans, uncached — OPEN
**Files:** `app/services/analyticsService.ts:96-144`, `app/services/adminService.ts:75-91`

`getLearningGapReport()` runs three whole-platform scans (`users`, `collectionGroup('scores')`,
`collectionGroup('gameStats')`) in the browser on every `AdminAnalyticsTab` mount/refresh;
`getPlatformStats()` does a full `users` + `gameStats` scan per dashboard load. At ~1,000
users this is tens of thousands of reads per admin page view.
**Fix:** move aggregation into an Admin-SDK route, materialize into `analytics/_report` on
the recompute cron, serve one cached read.

### P3. Multi-MB cover PNGs on the library grid — OPEN
**Files:** `public/games/*/cover.png`, `app/components/GameCard.tsx:30-35`, `next.config.js:5`

`code-quest/cover.png` ≈ 5.5 MB, `dialed-sound/cover.png` ≈ 3.7 MB, served raw
(`images.unoptimized: true`) across ~30 cards. Dwarfs round 1's #10.
**Fix:** convert covers to resized WebP (~400px, ~30–60 KB) and/or enable Next image
optimization.

### N5. Leaderboard recompute job is never invoked — OPEN
**Files:** `app/api/internal/leaderboards/recompute/route.ts`, `app/api/leaderboard/route.ts:65-93`

The recompute route exists but no cron/startup call invokes it anywhere in the repo, and
the read paths serve cached docs whenever they merely `exists` (no TTL). Net effect: the
first request does one expensive full `collectionGroup` scan (round 1 #4) then those values
freeze. Also keeps round 1 #5 open (page still does ~50 per-profile reads).
**Fix:** add a cron/systemd timer for the recompute route + a staleness TTL on the read
paths so a missed run degrades to recompute instead of serving stale-forever data.
