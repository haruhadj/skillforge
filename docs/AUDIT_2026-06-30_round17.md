## P-A — Vocab `ORDER BY RANDOM()` → cached candidate pools + JS sampling — ✅ FIXED
**Files:** `server/wordnet/service.js`, `server/wordnet/sample.js` (new)

**Verified:** `getWordsByDifficulty` (:116), `getDistractors` (:178) and `generateWordPairs`
(:356) each ran a filtered join with `ORDER BY RANDOM()` over the ~152k-word WordNet SQLite on
the synchronous event-loop thread. A single `/words?count=10` request fanned out into 10–15+ of
these full-table sorts (the code itself rate-limited those endpoints to 20/min as a band-aid).

**Fixed:** each query now runs **once per difficulty per process** (the DB is opened read-only and
never changes), caching the fully-filtered candidate rows in a module-level `Map`; every request
draws a random subset in JS via the pure `sampleDistinct(pool, n)` (extracted to `sample.js` so it
is unit-testable without `node:sqlite`). The heavy `ORDER BY RANDOM()`/`LIMIT` are gone; the SQL
filters are otherwise byte-for-byte identical, so word selection is unchanged. `getDistractors`
caches the pool **without** the per-call exclusion and drops the correct word at sample time.
**Measured:** first call 202 ms (warm the pool) → subsequent calls **0 ms**; draws still vary and
distractors still exclude the prompt word. Both live servers (`server/games/vocab/server.js`,
`server/games/spelling-bee/server.js`) import this canonical service; the unused `games-src`
copy was confirmed out of the live path.

## P1/P2 — admin analytics + activity feed: client scans → cached server routes — ✅ FIXED
**Files (new):** `app/api/admin/platform-stats/route.ts`, `app/api/admin/learning-gap/route.ts`,
`app/api/activity/route.ts`, `app/lib/requireAdmin.ts`, `app/lib/cacheFreshness.ts`,
`app/services/analyticsCompute.ts`
**Files (edited):** `app/services/adminService.ts`, `app/services/analyticsService.ts`,
`app/services/gameDataService.ts`, `firestore.rules`

**Verified:** the leaderboard already had a server cache (`/api/leaderboard` + recompute, 5-min
TTL), but three paths still ran client-side `collectionGroup` scans on every render:
`getPlatformStats()` (Dashboard — all gameStats), `getLearningGapReport()` (Analytics — all
users + all scores + all gameStats), and `getGlobalRecentActivity()` (public activity — all
gameStats, sorted, to return 50).

**Fixed:** all three now sit behind cached server routes that **mirror the existing
`/api/leaderboard` pattern** (a Firestore cache doc with `recomputedAt` + lazy compute-on-stale-read;
shared `isFresh`/`toMillis`/`uidFromPath` helpers extracted to `cacheFreshness.ts`):
- **`/api/admin/platform-stats`** and **`/api/admin/learning-gap`** are **admin-gated** by the new
  `requireAdmin()` (verifies the Firebase ID token via Admin SDK, then checks
  `users/{uid}.role == 'admin'`, like the score route's token model). 5-min TTL; cache docs in
  `adminCache/*`. The learning-gap math moved verbatim into the pure, unit-tested
  `computeLearningGapReport()` (no Firebase import) so the route fetches rows via Admin SDK and the
  algorithm stays identical.
- **`/api/activity`** is **public + IP-rate-limited** (like the leaderboard read); 60-second TTL;
  cache doc `activityCache/recent`.

The client service functions keep their old names/signatures (so components are unchanged) but now
just `fetch()` the routes — `getPlatformStats`/`getLearningGapReport` attach the caller's ID token;
`analyticsService` re-exports the types + `SUBJECT_COLORS`/`GAME_SUBJECTS` from `analyticsCompute`
so existing imports resolve. `firestore.rules` adds `adminCache` and `activityCache` as
**Admin-SDK-only** (`allow read, write: if false`) — the learning-gap report embeds PII-derived
at-risk-student emails, so clients reach it solely through the admin-gated route.

## P3 — WebP game covers — ✅ FIXED
**Files (new):** `scripts/optimize-covers.mjs`, `app/components/GameCover.tsx`
**Files (edited):** 6 render sites (`GameCard.tsx`, `AdminGamesTab.tsx`, `activity/page.tsx`,
`games/[gameId]/page.tsx`, `profile/page.tsx`, `profile/[uid]/page.tsx`), `package.json`,
generated `public/games/*/cover.webp`

**Verified:** 30 `public/games/<id>/cover.png` totalling **15.9 MB** (code-quest 5.3 MB +
dialed-sound 3.6 MB = 55%), all rendered via plain `<img src=".../cover.png">` hardcoded in 9 places.

**Fixed:** `npm run covers:optimize` (`scripts/optimize-covers.mjs`) generates a sibling
`cover.webp` for each PNG (max-width 768, quality 80). The path convention is centralized in a new
`<GameCover>` component — a `<picture>` (with `display:contents`, so it adds no layout box and the
`<img>` keeps each caller's exact classes) that serves WebP with a **PNG fallback**, so nothing
breaks if a `.webp` is missing. All 9 `<img>` sites now use it. **Result: 15.9 MB → 997 KB (−94%).**
The `.webp` files are committed; the Docker image copies `public/` verbatim (no Docker change).

> **Tooling note:** `sharp` is not resolvable in this repo's `node_modules`, so the script
> auto-falls back to the ImageMagick `magick` CLI (present, libwebp 1.5) — which generated the
> committed files. The script prefers `sharp` if/when installed; output is equivalent either way.
