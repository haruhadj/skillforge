# SkillForge — Audit Report (Round 15)

*Audited: 2026-06-30 · Findings verified against the codebase before recording.*

> Follow-up to rounds 1–14. This is the **multiplayer-integrity** round: the Chess server
> trusted the client for board state, so a malicious player could forge any position and the
> server would broadcast it to the opponent as truth. Named as the open item ("Chess server-side
> move legality") since round 11, and explicitly flagged in the code itself
> (`socket-server.js:222-227`, "audit M3").
>

## HIGH — Fixed this round (head-to-head integrity)

### R15-1. `move` handler trusted the client FEN → board forgery — ✅ FIXED
**Files:** `server/games/chess/moveValidator.js` (new), `server/games/chess/socket-server.js`

**Verified:** the `move` handler enforced *sender auth* (only a seated player may move,
`socket-server.js:220`) and *turn ownership* (the stored FEN's active color must match the
mover's color, `:228-231`), but then stored the client's FEN verbatim:
`room.fen = isValidFen(fen) ? fen : START_FEN`. `isValidFen` (`:113`) only checks the string has
six space-separated fields — **not** chess legality. So any seated player could
`emit('move', { fen })` with an arbitrary but structurally-valid FEN on their own turn — teleport
pieces, delete the opponent's pieces, spawn queens, jump from a lost to a won position — and the
server broadcast it to the opponent. (Round-9/11 server-authoritative scoring means this is a
fair-play / desync exploit between opponents, not leaderboard inflation.)

**Fixed:** the server now **recomputes** the board with `chess.js`. The pure
`applyMove(fen, move)` (`moveValidator.js`) loads the current (pre-move) FEN, replays the client's
`{ from, to, promotion }` move, and returns a **server-computed** FEN — the client's `fen` field
is now ignored entirely. An illegal/forged move is dropped (`if (!outcome.ok) return`) and never
broadcast; a legal move broadcasts `{ move, fen: room.fen }` with the engine's FEN. The prebuilt
client emits exactly `{ from, to, promotion }` and is itself chess.js-based, so the engines agree
and honest moves are never falsely rejected.

### R15-2. Null move ("pass") accepted → turn-skip exploit — ✅ FIXED (found during impl)
**File:** `server/games/chess/moveValidator.js`

`chess.js` treats `chess.move(null)` as a **null move** (a legal engine "pass", `san: '--'`), which
is not a legal move in real play. Left unguarded, a client could `emit('move', { move: null })` to
skip its own turn. `applyMove` now requires a concrete move (`{ from, to }` object or a SAN string)
up front and rejects any result whose `san === '--'`.

### R15-3. Server-authoritative game end (checkmate / stalemate / draw) — ✅ ADDED
**File:** `server/games/chess/socket-server.js`

Previously the game only ended server-side on `resign` / agreed `draw`; checkmate and draws were
purely client-driven, so a checkmated client could keep emitting moves. The engine now reports
`isCheckmate()` / `isStalemate()` / `isInsufficientMaterial()` / `isThreefoldRepetition()`, and the
handler emits the **existing** `gameEnded` shape (same as resign/draw) with the correct winner.
Additive and idempotent — the `room.gameEnded` guard blocks any further moves.

### R15-4. `START_FEN` was a non-standard position — ✅ FIXED
**File:** `server/games/chess/socket-server.js:11`

`START_FEN` was `rn1qkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR …` — black's back rank is missing the
c8 bishop. Latent (the client supplied its own FEN), but now that the server computes from this
base on a fresh room it must be correct: fixed to the standard
`rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1`.

## Fresh sweep — checked, no new finding
- Sender-auth (`room.players[socket.id]`) and turn-ownership pre-checks — kept as cheap
  defense-in-depth ahead of the engine (which also enforces turn via the FEN's active color). Clean.
- `offerDraw` / `respondDraw` / `cancelDrawOffer` / `resign` — all already gated on
  `room.players[socket.id]` and `room.gameEnded`; unchanged, no FEN trust. Clean.
- `move` throttling — intentionally still unguarded: turn-ownership bounds a socket to one move at
  a time (its next move is rejected until the opponent replies), so move-spam is naturally limited
  and the chess.js parse cost per move is negligible. Clean.
- DoS caps (rounds H1): room/socket/IP limits, idle reaper, per-socket burst guard — untouched. Clean.
