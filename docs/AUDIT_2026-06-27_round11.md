# SkillForge — Audit Report (Round 11)

*Audited: 2026-06-28 · Findings verified against the codebase before recording.*

> Follow-up to rounds 1–10. This is the **S2-b round** that rounds 9 and 10 named as the
> next headline: locking `users/{uid}/gameStats/{gameId}` so the client SDK can no longer
> write the leaderboard's inputs.
>
> Round 9 made `scores` server-authoritative, but `gameStats` stayed client-writable
> because the same doc carries the per-game **resume/progress blob** that `REQUEST_PROGRESS`
> restores. The leaderboard trusts that doc: `aggregateGlobalLeaderboard` and
> `aggregateGamePopularity` (`app/services/scoring.ts`) read `totalMatchCount` from it, and
> `computeCompositeScore` turns it into the +10 experience factor and the popularity
> ranking. So any authed user could `setDoc` an arbitrary `totalMatchCount` straight into
> their own `gameStats` and inflate their composite/tier and any game's popularity —
> bypassing the round-9 score lock.
>
> This round moves the resume blob behind a verified, Admin-SDK **server progress API**
> (`/api/games/progress`), namespaced under a `progress` field so the client can never
> reach the top-level weighted-stats fields, then **locks `gameStats` writes in the rules**.
> A latent rules-structure gap surfaced and is closed along the way (see R11-3).
>
> Scope: app code + one Firestore-rules change (**needs a rules deploy**) + a prod rebuild
> (the new route ships in the frontend image). **No game-client artifacts, no `server/`
> changes.**

## HIGH — Fixed this round (S2-b — server-authoritative gameStats)

### R11-1. Resume blob was written to `gameStats` via the client SDK — migrated to a server progress API — ✅ FIXED
**Files:** `app/api/games/progress/route.ts` (new), `app/lib/progress.ts` (new),
`app/play/[gameId]/PlayGameClient.tsx`, `app/services/gameDataService.ts`

**Verified:** the generic `GAME_STATS` branch of the host bridge called
`saveGameStats(uid, gameId, progress)` — a client-SDK `setDoc` to
`users/{uid}/gameStats/{gameId}` — and `REQUEST_PROGRESS` read it back with
`getGameStats`. Round 9 routed the *score* through `/api/games/score` but deliberately left
this client write in place (tagged `S2-b`) because the blob is opaque per-game resume
state. While that write existed, `gameStats` had to stay client-writable, and the
leaderboard-relevant fields in the same doc were forgeable.

**Fixed:** added `POST/GET /api/games/progress`, mirroring the hardened `/api/games/score`
pattern — Bearer ID token → `uid` (never from the body), `defaultGames` gameId allowlist,
per-uid 60/min throttle. The blob is validated by a pure, unit-tested helper
(`app/lib/progress.ts`: `sanitizeProgressBlob` — plain-object + 64 KB UTF-8 cap) and
stored **namespaced** as `gameStats.progress` via the Admin SDK. Namespacing is the key:
even a blob key literally named `totalMatchCount` lands at `gameStats.progress.totalMatchCount`,
never the top-level field the leaderboard reads — the client cannot forge leaderboard
inputs through this route. `GET` returns `data.progress`, falling back to a legacy
top-level blob (`extractLegacyProgress`, top-level keys minus the weighted-stats keys) so
existing in-flight resume survives the migration. The host bridge now uses fire-and-forget
`postProgress` / `fetchProgress` helpers (a failed progress call never blocks gameplay).
The now-dead `saveGameStats` / `getGameStats` / `saveModeScoreStats` client writers were
removed from `gameDataService.ts` (reads — `getAllGameStats`, recent-activity — stay).

### R11-2. `gameStats` was client-writable — locked read-only in the rules — ✅ FIXED (needs rules deploy)
**File:** `firestore.rules`

**Verified:** `match /users/{uid}/gameStats/{statsId}` granted `read, write` to the owner,
so a direct client-SDK `setDoc` could set any `totalMatchCount` / weighted-stats field.

**Fixed:** dropped the client write — `gameStats` is now owner-**read-only**, exactly like
`scores` (R9-2). The only write paths are `/api/games/score` (weighted stats) and
`/api/games/progress` (resume blob), both Admin SDK (which bypass rules after verifying the
token). Owner read + the `collectionGroup('gameStats')` leaderboard read are unchanged.

**Deploy:** `firebase deploy --only firestore:rules --project skillforge-7a058`

### R11-3. Recursive owner-subcollection fallback re-granted writes — narrowed to read-only — ✅ FIXED (needs rules deploy)
**File:** `firestore.rules`

**Verified:** Firestore grants an operation if **any** matching `match` block allows it
(rules are OR'd — there is no "most specific wins"). The fallback
`match /users/{uid}/{sub}/{document=**}` had `allow read, write`, and that pattern **also
matches** `scores/{gameId}` and `gameStats/{statsId}`. So removing `write` from the
specific matches is not enough — the fallback re-grants it. This means R9-2's `scores`
lock was effectively reopened by this fallback, and a naive R11-2 would have been too.

**Fixed:** narrowed the fallback to `allow read` only. A grep of `app/` confirms the **only**
client-SDK subcollection writes under `users/{uid}` are `scores` and `gameStats` (both now
server-only); `recentlyPlayed` is a field on the **root profile doc**
(`userProfileService.ts`), governed by `match /users/{uid}`, not a subcollection. So the
fallback's `write` was unused by any client path. This also **retroactively completes
R9-2** — a client `setDoc` to `scores/{gameId}` is now genuinely denied.

## Fresh sweep — checked, no new finding
- `/api/games/progress` — verified-token uid, registry-checked gameId, plain-object +
  64 KB blob cap, per-uid throttle, Admin-SDK namespaced write. Symmetric with
  `/api/games/score`. Clean.
- `buildWeightedModeStats` write path (`/api/games/score`) — still the sole writer of the
  top-level weighted stats; unaffected by the progress namespacing (`merge:true`, disjoint
  keys). Clean.
- `getRecentActivity` / `getGlobalRecentActivity` — still order by top-level `updatedAt`;
  the progress route sets it, so progress-only games keep appearing in the feed
  (behavior preserved). Clean.
- `firestore.rules` — `scores` + `gameStats` now both owner-read / server-write; root
  `role` protection (rounds 2/7) and collectionGroup reads unchanged. Clean.
