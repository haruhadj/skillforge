## HIGH — Fixed this round (PII exposure)

### R14-1. `users/{uid}` was world-readable → every user's email/PII exposed — ✅ FIXED (needs rules deploy)
**Files:** `firestore.rules`, `app/services/publicProfileService.ts` (new),
`app/lib/publicProfileFields.ts` (new), plus all cross-user read sites (below).

**Verified:** `firestore.rules` had `match /users/{uid} { allow read: if true }`. Firestore
rules cannot field-filter a read, so the **entire** profile doc leaked to any client:
`email`, `linkedProviders.*.email` (per-provider OAuth emails), `role` (admin status), and the
device-fingerprint fields (`deviceType/Os/Browser/LastSeen`). `adminService.ts:34`'s
`getAllUsers()` is a plain `getDocs(collection('users'))` with no rule gating it — so **any
client could enumerate every user's email**. The public profile viewer also deliberately
rendered another user's email to any visitor (`profile/[uid]/page.tsx:149-151`).

**Fixed:** introduced a minimal, world-readable **`publicProfiles/{uid}`** collection holding
ONLY display fields (`username`, `usernameNormalized`, `photoURL`, `photoThumbURL`). The
PII-bearing `users/{uid}` doc is now **owner+admin read only**:
```
allow read: if request.auth != null && (request.auth.uid == uid || isAdmin());
```
`isAdmin()` (rules 6-9) uses `get()`, which is **exempt from read rules**, so the admin check
itself is unaffected; admins can still `list` the users collection (the rule is uid-independent),
non-admins cannot — closing the mass email-enumeration hole. The public profile viewer no
longer shows other users' email.

### R14-2. `publicProfiles` write integrity — `hasOnly` key allowlist — ✅ FIXED (needs rules deploy)
**Files:** `firestore.rules`, `app/lib/publicProfileFields.ts`

The public doc is owner-written (the values are user-controlled display data the user already
client-writes to their own profile), but the rule constrains it so PII can **never** land there:
```
allow create, update: if request.auth != null && request.auth.uid == uid &&
  request.resource.data.keys().hasOnly(
    ['username','usernameNormalized','photoURL','photoThumbURL','updatedAt']);
```
The pure `pickPublicProfileFields()` helper (`app/lib/publicProfileFields.ts`) enforces the
same allowlist client-side and is the single source of truth, unit-tested to prove
`email`/`role`/`linkedProviders`/`device*` are stripped even when handed a full `UserProfile`.

### R14-3. Cross-user reads migrated to `publicProfiles`; display fallback hardened — ✅ FIXED
**Files:** `app/leaderboard/page.tsx`, `app/games/[gameId]/page.tsx`, `app/activity/page.tsx`,
`app/profile/[uid]/page.tsx`, `app/services/userProfileService.ts`

Every **cross-user** `getUserProfile(uid)` call now uses `getPublicProfile(uid)`; the
`email?.split('@')[0]` display fallback (which relied on the leaked email) is dropped in favor
of `username || 'Unknown'`/`'Player'`. **Own-user** reads (`ensureUserProfileDocument`,
`TopNav`, `/profile`) keep `getUserProfile` (owner read still allowed). `syncPublicProfile()`
mirrors display fields into `publicProfiles` on all three owner write paths
(`ensureUserProfileDocument`, `claimUsername`, `uploadProfilePhoto`) — self-healing on every
login, fire-and-forget so a mirror failure never blocks sign-in.

### R14-4. One-time backfill for existing users — ✅ ADDED
**File:** `scripts/backfill-public-profiles.mjs`

Admin-SDK script that mirrors every existing `users/{uid}` display field into
`publicProfiles/{uid}` (idempotent `merge`, batched at 450). Run BEFORE the rules deploy so
existing users keep their names/avatars on the leaderboard.
