# SkillForge — Audit Report (Round 12)

*Audited: 2026-06-29 · Findings verified against the codebase before recording.*

> Follow-up to rounds 1–11. This round closes the two remaining security items from the
> round-11 backlog: **photoURL scheme/domain restriction** (deferred from R9-3) and the
> first phase of the **CSP rollout**.
>
> The attack surface: every OAuth provider returns a photo URL that was copied, unvalidated,
> into Firebase Auth and Firestore — from there it reached `<img src>` and `<AvatarImage src>`
> in seven+ locations across the app. A malicious provider (or a tampered callback) could
> inject a `javascript:` URL or an attacker-controlled image endpoint into every avatar
> render. Additionally, no Content-Security-Policy header existed at all; `object-src`,
> `base-uri`, and `frame-ancestors` are always safe to add and were the obvious gap.
>
> Full `script-src` / `connect-src` / `style-src` CSP is deferred: game iframes run
> pre-built Vite bundles (likely need `unsafe-eval`) and Firebase websockets span many
> subdomains — those need a dedicated audit pass before they can be safely constrained.
>
> Scope: two new source files (`sanitizePhotoURL.ts`, `sanitizePhotoURL.test.ts`) + targeted
> edits to `oauthLinks.ts`, `userProfileService.ts`, and `next.config.js`. **No Firestore
> rules change, no game server changes, no deploy of rules needed.**

---

## HIGH — Fixed this round

### R12-1. photoURL accepted from OAuth providers without scheme or domain validation — ✅ FIXED

**Files:** `app/lib/sanitizePhotoURL.ts` (new), `app/lib/oauthLinks.ts`,
`app/services/userProfileService.ts`

**Verified:** `backfillProfile()` in `oauthLinks.ts` (line 71) passed the OAuth-supplied
`p.photoURL` directly to `adminAuth.updateUser()`, and `createUser()` (line 124) passed it to
`adminAuth.createUser()` — both without any validation. `userProfileService.ts` (lines 295-297)
copied `user.photoURL` to the Firestore profile doc without validation. The value then reached
`<AvatarImage src>` and `<img src>` in TopNav, profile pages, leaderboard, activity feed, and
admin panel — seven+ render sites. A `javascript:` scheme or an attacker-controlled domain
could enter via any OAuth callback.

**Fixed:** new pure utility `app/lib/sanitizePhotoURL.ts` exports `sanitizePhotoURL(url)`:

- Falsy → `null`
- `data:image/` prefix → pass-through (base64 uploads from `uploadProfilePhoto` are safe)
- Malformed URL → `null`
- Scheme not `https:` → `null`
- Hostname must match an explicit allowlist of OAuth photo CDNs:
  `*.googleusercontent.com`, `avatars.githubusercontent.com`, `pbs.twimg.com`,
  `*.twimg.com`, `graph.facebook.com`, `*.fbcdn.net`, `*.fbsbx.com`,
  `*.tiktokcdn.com`, `*.tiktokcdn-eu.com`
- Anything else → `null` (triggers avatar initial fallback in the UI)

Applied at **write time** — the two Admin SDK callsites in `oauthLinks.ts` and the Firestore
write in `userProfileService.ts`. No render-site changes needed: once the stored value is
clean, every downstream `src` is trusted. Defense-in-depth at render time is out of scope
for this round.

**Tests:** `tests/sanitizePhotoURL.test.ts` — 15 cases covering valid URLs for each
provider, `data:image/` pass-through, `http:` rejection, `javascript:` rejection, `data:text/`
rejection, arbitrary domain rejection, lookalike domain rejection, malformed URL rejection,
null/undefined/empty input.

### R12-2. No Content-Security-Policy header — partial CSP added — ✅ FIXED (safe subset)

**File:** `next.config.js`

**Verified:** the existing `headers()` block had X-Frame-Options, X-Content-Type-Options,
Referrer-Policy, and HSTS — but zero CSP. `object-src`, `base-uri`, and `frame-ancestors`
have no false-positive risk and were missing entirely.

**Fixed:** added a fifth header with the safe-to-ship subset:

```
Content-Security-Policy:
  default-src 'self';
  object-src 'none';
  base-uri 'self';
  frame-src 'self';
  frame-ancestors 'self';
  img-src 'self' data: blob:
    https://*.googleusercontent.com
    https://avatars.githubusercontent.com
    https://pbs.twimg.com https://*.twimg.com
    https://*.fbcdn.net https://*.fbsbx.com
    https://graph.facebook.com
    https://*.tiktokcdn.com https://*.tiktokcdn-eu.com
```

- `object-src 'none'` — eliminates the Flash/plugin attack surface
- `base-uri 'self'` — prevents base-tag injection from changing all relative links
- `frame-src 'self'` — same-origin game iframes under `/games/*` still work; cross-origin
  embeds blocked
- `frame-ancestors 'self'` — supersedes `X-Frame-Options` in modern browsers; both are kept
  for legacy compatibility
- `img-src` — mirrors the `sanitizePhotoURL` allowlist exactly, so even if a future code
  path bypasses the write-time sanitizer, the browser itself refuses to load the image

**Deliberately deferred** (full CSP round needed):
- `script-src` — game iframes are pre-built Vite bundles; `unsafe-eval` usage must be audited
- `connect-src` — Firebase websockets span many subdomains; needs enumeration
- `style-src` — Tailwind emits inline styles

## Fresh sweep — checked, no new finding

- `app/api/auth/callback/*/route.ts` (google, github, twitter, tiktok, facebook) — these
  build an `OAuthProfile` object and call `resolveSignInUid(p)`, which feeds into the now-
  sanitized `backfillProfile()` and `createUser()` paths. No additional callsite for
  `photoURL` exists outside `oauthLinks.ts`. Clean.
- `uploadProfilePhoto()` in `userProfileService.ts` — converts user-uploaded files to
  `data:image/` base64 strings before writing to Firebase Auth and Firestore. These are
  self-contained; `sanitizePhotoURL` passes them through unchanged. Clean.
- Render sites (TopNav, profile pages, leaderboard, activity, games, admin) — all read
  `photoURL` / `photoThumbURL` from Firestore. Now that the stored value is sanitized at
  write time, these are safe without per-site changes. The `img-src` CSP adds browser-level
  enforcement as defense-in-depth. Clean.
