## CRITICAL — Fix Immediately

### 1. Unauthenticated Admin Delete-User API — ✅ DONE (2026-06-26)
**File:** `app/api/admin/delete-user/route.ts`

**Verified:** the POST handler read `uid` from the request body and performed no
authentication or authorization. Any unauthenticated HTTP request with a valid
UID could permanently delete any user's account and all their data — direct
privilege escalation.

**Fixed:** the route now requires a `Bearer` ID token, verifies it via
`getAdminAuth().verifyIdToken()`, and confirms the caller's user doc has
`role === 'admin'` (the project's actual admin marker — `role`, not `isAdmin`)
before deleting the target uid. Unauthenticated → 401, non-admin → 403.
The caller `app/components/AdminUsersTab.tsx` (`handleDeleteUser`) was updated to
send `Authorization: Bearer <currentUser.getIdToken(true)>`. Typecheck passes.

### 2. Hardcoded Secret in `.env.example` — ✅ DONE (2026-06-26)
**File:** `.env.example` line 92

**Verified:** `INTERNAL_API_SECRET=Haruhadj23@` — a live value sitting directly
under a "Generate with: openssl rand -hex 32" comment.

Rotate this secret in production now, then replace the example value with a
placeholder such as `<generate with: openssl rand -hex 32>`.

## HIGH — Address Soon

### 3. OAuth Secrets Baked into Docker Image Layers
**File:** `docker-compose.prod.yml` (lines 34–42 under `args:`)

**Verified:** all five `*_OAUTH_CLIENT_SECRET` values are passed as Docker build
`args:`, which get permanently embedded in image layer history (`docker history`
exposes them). They are *also* already provided at runtime under `environment:`
(lines 53–61), so the fix is simply to **delete them from the `args:` block**.

### 4. Leaderboard Fallback Does a Full Firestore Scan
**File:** `app/api/leaderboard/route.ts`

**Verified:** on a `leaderboards/_global` cache miss, the fallback runs
`collectionGroup('scores')` + `collectionGroup('gameStats')` scans across every
user — scales linearly with user count (~31k+ reads at 1,000 users).

**Fix:** ensure `/api/internal/leaderboards/recompute` runs after every deploy
and on a cron schedule (every 4–6 hours).

### 5. Leaderboard Page Makes ~50 Separate Profile Reads
**File:** `app/leaderboard/page.tsx`

**Verified:** `TOP_COUNT = 50` and `fetchProfiles()` calls `getUserProfile(uid)`
once per uid (50 individual Firestore reads).

**Fix:** denormalize `username` and `photoURL` into the
`leaderboards/_global.entries` array so the page needs only 1 read total.
