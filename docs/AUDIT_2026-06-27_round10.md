## MEDIUM — Fixed this round

### R10-1. Socket per-IP throttle was bypassable via a spoofed X-Forwarded-For — ✅ FIXED
**Files:** `server/games/chess/socket-server.js`, `server/games/tictactoe/socket-server.js`,
`server/games/chroma-memory/socket-server.js` (identical `clientIp()`)

**Verified:** the per-IP connection throttle (`MAX_CONN_PER_IP = 30`, round 6's H1 fix) keys
its sliding window on `clientIp(socket)`, which read the **first** entry of `X-Forwarded-For`
(`String(xff).split(',')[0]`). But nginx fronts these via `$proxy_add_x_forwarded_for`, which
**appends** the real peer as the **last** entry — so the first element is whatever the client
sent. An attacker rotating a fake first hop (`X-Forwarded-For: 1.2.3.4`, `5.6.7.8`, …) lands in
a fresh throttle bucket on every handshake, defeating the cap and re-opening the global
room-exhaustion DoS the round-6 caps were meant to bound.

**Fixed:** parse the **last** XFF element — the hop nginx itself observed and appended.
`docker-compose.prod.yml` only `expose`s these servers (no published port), so exactly one
trusted proxy (nginx) ever sets the header and the last hop is the real, non-forgeable client
IP. Identical three-line change in all three servers; falls back to `socket.handshake.address`
when the header is absent (direct/dev connections).

### R10-2. Vocab heavy endpoints were query-amplifying under a too-generous limiter — ✅ FIXED
**File:** `server/games/vocab/server.js`

**Verified:** `/api/vocab/words` and `/api/vocab/wordpairs` each fan out into dozens of
`ORDER BY RANDOM()` full-scan joins over the large WordNet SQLite DB, executed synchronously on
the event-loop thread (N+1: one distractor query per word). Both shared the flat
`apiLimiter` (120 req/min) used for cheap endpoints like `/health` and `/stats`, and accepted
`count` up to 25 (words) / 50 (wordpairs) — so a single authenticated-or-not client could issue
120 maximally-amplifying requests per minute and pin CPU/IO.

**Fixed:** added a dedicated `heavyLimiter` (20 req/min) mounted on just those two routes —
mirroring the pattern spelling-bee already uses for its TTS endpoint — and tightened the per-
request `count` clamps to **15** (words) and **30** (wordpairs). The global `apiLimiter` still
covers the cheap routes. This bounds the per-request and per-minute cost within the deployment
topology without touching the query logic.

> **Deferred (not this round):** rewriting `ORDER BY RANDOM()` → sampled-id lookups. That would
> touch game-facing query behaviour; the limiter + clamp mitigates the DoS in the meantime.

## LOW / INFO — Fixed this round (defense-in-depth)

### R10-3. TTS cache had no key cap — ✅ FIXED
**File:** `server/games/spelling-bee/server.js`

**Verified:** the Edge-TTS audio cache (`new NodeCache({ stdTTL: 600, checkperiod: 120 })`) is
keyed on `${mode}:${text}` with no `maxKeys`. Distinct large texts within the 10-minute TTL
grow memory unbounded.

**Fixed:** added `maxKeys: 1000`. NodeCache throws on insert past the cap, so the `cache.set`
is now wrapped in try/catch — a full cache degrades to "compute-and-serve without caching"
instead of erroring the request.

### R10-4. No per-socket event-flood guard; uncapped client roomId — ✅ FIXED
**Files:** the 3 socket servers

**Verified:** beyond the per-IP *connection* throttle, a single established socket could emit
spam-prone events (`joinRoom`/`offerDraw`/`cancelDrawOffer` in chess; `create_room`/`join_room`/
`request_rematch` in tictactoe; `create_room`/`join_room`/`rematch_to_lobby` in chroma) with no
per-socket rate limit, and the client-supplied `roomId`/`roomCode` was used as a Map key /
processed with no length cap.

**Fixed:** added a shared `tooFast(socket)` helper — a generous per-socket token bucket (50
events / 5 s, well above any legitimate cadence including blitz play) — gated at the top of the
spam-prone emitters only. Per-move emitters are deliberately **not** tightly gated (round-6 turn
guards already bound those and fast play is legitimate). Also added a `MAX_ROOM_ID_LEN = 64`
reject on the client-supplied room id before it becomes a Map key.

### R10-5. REST CORS origin lists had drifted — ✅ FIXED
**Files:** `server/games/vocab/server.js`, `server/games/hamaru/server.js`,
`server/games/spelling-bee/server.js`

**Verified:** the three REST servers' production `origin` allowlists had diverged — vocab was
missing the real prod domain `https://skillforge.haruhadj.org` (per CLAUDE.md) and hamaru was
missing `https://haruhadj.github.io`. Functionally moot today (the games call these same-origin
through nginx, so no CORS preflight), but it's latent config drift.

**Fixed:** aligned all three to the same canonical prod set —
`['https://skillforge.haruhadj.org', 'https://skillforge.haruhadj.duckdns.org', 'https://haruhadj.github.io']`.
